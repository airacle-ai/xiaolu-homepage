## 一、定位框架：五层价值模型

```
┌──────────────────────────────────────────────────┐
│  第五层：行动层                                    │
│  每个洞察配一个微小行动，"下次试试这个"             │
│  让产品从镜子变成练习场                            │
├──────────────────────────────────────────────────┤
│  第四层：留存层（时间轴）                           │
│  "我变了"——三个月后重测，身份标签的变化             │
│  "关系变了"——同一对关系的变化追踪                  │
├──────────────────────────────────────────────────┤
│  第三层：传播层（关系碰撞）                         │
│  关系类型命名 + 可分享的关系名片                    │
│  传播语言 ≠ 产品语言 ≠ 获客语言                    │
├──────────────────────────────────────────────────┤
│  第二层：可信度层（结构化维度 + 证据层）             │
│  19 维结构化 + 三级证据展开 + "我不认同"修正出口    │
│  让用户从"AI胡说"到"它比我想的还清楚"              │
├──────────────────────────────────────────────────┤
│  第一层：被理解感（AI 对话）                        │
│  不是测试，是聊天。不是分析，是有人真的在听你说话。   │
│  "aha moment"密度 × 即时反馈 × 矛盾洞察            │
└──────────────────────────────────────────────────┘
```

### 五层之间的关系

- **被理解感**是入口——没有这一层，后面四层没人会走到
- **可信度**是转化——用户从"说得挺像"变成"这真的有依据"，才会愿意分享
- **传播**是增长——但传播的前提是用户对自己的身份标签有认同感（被理解感 + 可信度共同支撑）
- **留存**是生命线——没有这一层，传播拉来的人也会流失
- **行动**是差异化——所有人格测试止于"告诉你是什么"，只有你告诉用户"然后怎么做"

### 三层信息语言系统

| 场景 | 语言策略 | 例子 |
|------|----------|------|
| 获客信息 | 猎奇 + 社交货币 | "看看你和他是什么关系"、"测测你的隐藏人格" |
| 产品内 | 被看见 + 被理解 | "你有一种自动运行的共情能力" |
| 传播物料 | 身份标签 + 关系命名 | "我们是【镜像成长型】"、"我是调和者·深度型" |

这三个不一致是**故意的**。获客需要钩子，产品需要深度，传播需要简洁。试图用同一种语言覆盖三层会导致每一层都平庸。

## 二、被理解感：深度优化方案

### 2.0 被理解感的定义

被理解感 ≠ "AI说对了"。

被理解感 = 用户产生 **"我以为只有我自己知道这件事"** 的瞬间。

这个瞬间由三个要素构成：
- **命名**：把用户模糊的感受变成清晰的语言（"你刚才说的那个不舒服，可能是边界被侵犯的感觉"）
- **矛盾揭示**：指出用户自己没意识到的行为矛盾（"你说你重视自由，但在三个场景里你都选择了留在群体里"）
- **正常化**：让用户知道"你这样不是奇怪，是有原因的"（"这不是冷漠，你在避免无效消耗"）

### 2.1 当前方案的问题

当前 Oprah 的对话设计在 prompt 里已经有一个不错的基础——洞察框架、追问原则、说话风格都有雏形。但从产品视角看，缺了以下几个关键设计：

| 问题 | 当前表现 | 影响 |
|------|----------|------|
| 开场没有"钩子洞察" | 开场是自我介绍 + 直接出情境题 | 用户前 3 轮没有"被看见"的感觉 |
| aha moment 太晚 | 第一个有意义的洞察通常在第 6-8 轮 | 用户在前几轮可能流失 |
| 没有"你被听到了"的即时反馈 | 用户回答后，Oprah 直接追问或发下一个情境 | 用户不知道自己说的东西被认真处理了 |
| 结束没有"收束仪式" | 结果是跳转到另一页看地图 | 对话体验和结果体验是割裂的 |
| 没有处理"用户不信/不认同"的路径 | 用户觉得不准时只能放弃 | 沉默流失 |
| 对话节奏单一 | 全程一个节奏——提问→回答→追问 | 缺乏呼吸感 |

### 2.2 优化：五幕式对话结构

将当前平铺的提问-追问循环重构为有情绪弧线的五幕结构：

```
第一幕：入场（1-2轮）
  - 简短自我介绍
  - 一个"安全但能暴露信号"的开场问题
  - 用户回答后，给出第一个微洞察（命名型）
  - 目标：让用户在前3轮内产生第一次"被看见"

第二幕：展开（3-6轮）
  - 2-3个情境题 + 追问
  - 每2轮至少给一个洞察
  - 如果用户说了矛盾的话，立刻抓住
  - 目标：积累信号，建立信任

第三幕：转折（1-2轮）
  - 一个稍微深入的问题（触及被压抑的表达或未完成的自我）
  - 给一个较长的"阶段总结洞察"——综合前面几轮的发现
  - 目标：让用户感到"这个AI不只是问问题，它在思考我"

第四幕：收束（1-2轮）
  - 确认收束意愿
  - 给一句"总结式洞察"——用用户自己的语言风格来回应用户
  - 目标：让用户对"分析结果"产生期待

第五幕：揭晓（跳转地图页）
  - 不再是冷冰冰的数据展示
  - 而是一段像朋友在描述你的文字 + 身份标签
  - 目标：让结果页成为对话的延续，而非断裂
```

### 2.3 开场优化：前三轮的"被看见"设计

**当前开场（prompt中的示例）：**

> "嘿！我是 Oprah。接下来我们会聊一些有趣的问题，帮你更清楚地看见自己。没有对错，说你真实的想法就好。我们从一个场景开始——"

问题：这是标准的AI教练开场，没有个性，没有洞察，用户在回答第一个问题之前没有任何"这个AI可能真的懂我"的预期。

**优化后：三阶段入场**

**第 1 轮——Oprah 开场（建立人格 + 降低防御）：**

```
"嘿，我是Oprah。我不是什么心理专家，就是个对你这个人感到好奇的朋友。

我们聊天的节奏大概是这样：我问你一些场景问题，你来回答。过程中如果我注意到什么有意思的模式，我会直接告诉你——你可能会觉得'诶你怎么知道的'。

不用准备什么，真实的想法就好。准备好了吗？"
```

关键设计：
- "我不是心理专家"——降低被审视感
- "你可能会觉得'诶你怎么知道的'"——预设了 aha moment，制造期待
- 不用情境题直接开场，而是给了用户一个心理准备

**第 2 轮——第一个情境题（低威胁 + 高信号密度）：**

选择"情境4：完全属于你的两天"替代当前的"深夜的求助"作为开场题。原因是：

| 对比维度 | 深夜的求助（当前开场） | 完全属于你的两天（建议开场） |
|----------|----------------------|---------------------------|
| 威胁感 | 涉及社交责任和道德判断 | 纯个人偏好，没有对错 |
| 信号密度 | 3-4 个维度 | 4-5 个维度（含被压抑的表达） |
| 回答心理负担 | 高——"我是不是不够朋友" | 低——"我就说我真实想干嘛" |
| 洞察空间 | 局限于关系模式 | 打开生活方式 + 自我认知 |

**第 3 轮——第一个微洞察（命名用户没说出口的东西）：**

用户回答"完全属于你的两天"后，Oprah不说"好的，下一个问题"。而是给一个命名型洞察：

```
"有点意思——你描述的那个周末，其实不是'什么都不做'。你在给自己创造一个没有外界噪音的空间，然后在里面认真地做一些你真正在意的事。

我注意到一个模式：你跟自己相处的方式不是放空，是用独处来自我整理。这在社交能量上其实挺特别的——很多人独处是为了逃避社交，但你的独处像是在给自己充电然后更好地回去。

这种模式，在你的日常生活里也这么明显吗？"
```

这实现了被理解感的三个要素：
- **命名**（"用独处来自我整理"）——给了用户一个他之前可能没有的语言
- **正常化**（"这在社交能量上其实挺特别的"）——让用户觉得自己的模式被认可而不是被评判
- **矛盾揭示**（隐性的——"不是放空，是整理"——纠正了一个常见的误解）

### 2.4 对话中洞察的节奏控制

当前 prompt 里说了"不要每个回答都给洞察"，但没有明确的节奏规则。需要一条具体的规则：

**洞察密度规则：**

```
第 1-2 轮用户回答后 → 必须给 1 个微洞察（命名型）
第 3-5 轮               → 至少 1 个矛盾揭示型洞察
第 6-8 轮               → 每 2 轮至少 1 个洞察
第 9-12 轮              → 1 个阶段总结洞察（综合型）
收束前                  → 1 个收束洞察（呈现型——"在我眼中你是这样的..."）
```

**洞察类型切换策略：**

| 阶段 | 主要洞察类型 | 目的 | 示例 |
|------|-------------|------|------|
| 早期 | 命名型 | 建立"AI懂我"的信任 | "你刚才描述的其实是一种叫'框架构建'的思维方式——你先搭骨架，再填内容。" |
| 中期 | 矛盾揭示型 | 制造 aha moment | "你之前说看重自由，但在朋友群聊那个场景里，你选择了沉默。这两个放在一起看挺有意思的——" |
| 中后期 | 正常化 + 优势型 | 建立安全感 + 赋能 | "你倾向于优先保护关系的完整性，这在很多情况下其实是一种能力。但你还没练习过的是——不破坏关系的前提下说出不同意见。" |
| 收束 | 呈现型 | 制造对结果的期待 | "我大概能看到你的轮廓了。有几个地方我觉得特别有意思，等分析做完我想给你看——" |

### 2.5 即时反馈机制："你的话被听到了"

当前设计中，用户发送消息 → Oprah 回复下一段话。用户没有办法感知"我说的东西正在被处理"。

**优化方案：在 Oprah 的回复中增加"回声"**

不是复读用户的话——而是用 Oprah 的重新表述来确认理解：

```
当前做法：
用户："我可能会先睡，第二天再找他聊"
Oprah："有意思。那如果他没回你的'明天聊'，你第二天会主动找他吗？"

优化做法：
用户："我可能会先睡，第二天再找他聊"
Oprah："所以你是那种需要时间消化再回应的人——你不想在自己状态不好的时候给出一个半吊子的回应。
有意思。那如果他没回你的'明天聊'，你第二天会主动找他吗？"
```

回声的作用：
- 告诉用户"我听到的不只是你的决定，还有你决定背后的逻辑"
- 给用户的直觉一个更高层次的语言——用户可能没意识到自己是"需要时间消化再回应"
- 同时也是一种**隐性洞察**——不显式说"你是XX型"，但让用户隐约感到被看透

**回声的使用时机：**
- 每 2-3 轮用户回答后至少一次
- 不单独成段，嵌入在追问或过渡中
- 不超过一句话

### 2.6 矛盾捕捉与浮现

如果说命名是"告诉我我知道但说不出的东西"，矛盾揭示就是"告诉我我不知道的东西"。后者是被理解感最强的来源。

**当前问题：** prompt 里写了"好的洞察 = 用户没意识到但立刻觉得对的东西"，也写了追问到第二层的原则，但没有系统化的矛盾捕捉策略。

**优化：三层矛盾捕捉**

```
第一层：行为矛盾（言行不一）
  信号：用户说自己"很果断"，但回答选择题时反复犹豫
  呈现方式："你说你是个果断的人，但刚才那个场景里，你在两个选项之间来回了三分钟才决定。其实果断也许不是你现在的默认模式，而是你想成为的样子？"
  注意：语气不能是"你被抓到了"，而是"这其实挺有意思的"

第二层：情境矛盾（不同场景下的不一致）
  信号：用户在朋友场景中选择"直接表达不同意见"，在工作场景中选择"沉默"
  呈现方式："你对朋友挺直接的，但在工作群聊里你选择了不发声。这两个情况对你来说真正的差别是什么——是后果不同，还是你在不同圈子里其实是不同的人？"

第三层：价值矛盾（两个都想要但冲突了）
  信号：用户在道德困境中既想维护公平又不想伤害同事
  呈现方式："你现在卡在这里了——公平和关怀都对你很重要，但在这个情况下它们指着完全不同的方向。你通常是卡在这种地方，还是你内心其实有一个更倾向的选择？"
```

这三层矛盾并非每次对话都能覆盖，但 Oprah 需要在听到矛盾信号时**立刻切换追问策略**——放下原定流程，追这个矛盾。

### 2.7 处理"用户不信"的路径

当前产品没有处理负面体验的出口。用户觉得"不准"时，唯一的操作是关掉。

**需要设计三条出口路径：**

**路径 A：轻量修正（调整结果）**

在结果页每个维度旁加一个"这不完全对"按钮：

```
点击 → 弹出 3 个该维度的备选结果（由 LLM 在初次分析时预生成但未采纳）
     → 用户选择一个更接近的
     → 该维度更新，不影响其他维度
     → 同时记录"用户修正"，影响后续分析的倾向
```

**路径 B：定向重测（重新对话）**

用户觉得某个模块不准：
```
点击"重新探索这个方向" → 进入 3 轮定向对话 → 只更新该维度
```
这和当前的置信度提升机制类似，但入口是"不认同"而非"置信度不够"。

**路径 C：全局重来（重新开始）**

用户觉得整个画像都不对：
```
"完全重来" → 保留旧画像作为"版本1" → 重新对话 → 生成"版本2" → 可以对比
```
重来不是覆盖，而是版本化。用户可以看到"这两次的结果对比"，这本身也是一个洞察来源——"你两次的结果差别很大，可能是因为你这次更放松了？"

### 2.8 对话结束的仪式感：从对话到结果的过渡

当前过渡是冰冷的：用户点"查看结果" → loading 转圈 → 地图页。

**优化：增加"揭晓前预告"**

在 Oprah 的最后一条消息（标记 [ANALYSIS_READY] 的那条）中，不是简单说"你想看看结果吗"，而是：

```
"我们聊了不少了。在我眼中你的轮廓大概是这样的——

你是一个会在关系里认真调整温度的人，不喜欢用粗暴的判断去简化一个人。你做决定之前需要时间，但一旦想清楚了就会很坚定。你在学着用一种更直接的方式表达自己——不再为了不麻烦别人而过度压缩自己。

这只是粗略的印象。真正的分析结果里会有一些你可能自己都没注意到的模式。要现在看看吗？"
```

这实现了两件事：
- **在对话内就给了"呈现型洞察"**，让用户对结果产生期待
- **预披露了几个发现**，降低了结果的"突然感"——用户在点"查看结果"时已经有心理预期

### 2.9 被理解感优化的技术实施

这些优化主要落在 `src/lib/claude.ts` 的 `COACH_SYSTEM_PROMPT` 中。改写要点：

1. **增加五幕式结构描述**：在 prompt 中用清晰的章节描述每幕的目标和节奏
2. **增加洞察密度规则**：用具体的数字（"第1-2轮必须给1个命名型洞察"）代替模糊的"不要每个回答都给洞察"
3. **增加回声机制**：在对话策略中增加"每2-3轮在回复中先回声再追问"
4. **增加矛盾捕捉优先级**：明确写"当你发现矛盾时，停下原计划，优先追矛盾"
5. **增加收束仪式**：给出 [ANALYSIS_READY] 前必须输出呈现型洞察的规则

## 三、四个补位问题的解决方案

### 3.1 冷启动：单人也能玩

#### 3.1.1 问题本质

碰撞是双边行为——双方都需要完成自画像。早期用户的朋友大概率不在系统里。如果用户输入暗号 → 对方不存在 → 体验终止，用户会流失。

#### 3.1.2 方案：三种碰撞模式

**模式 A：双人完整碰撞（当前模式，精度最高）**

双方都完成自画像 → 输入暗号 → 生成完整碰撞分析。

适用：朋友已经在系统里。

**模式 B：单人推测碰撞（一方完成自画像 + 一方手动描述）**

用户手动为朋友创建一个"临时画像"：回答 5-6 个简化问题或直接选标签 → 生成碰撞分析 → 附带"邀请卡"发给朋友。

```
流程：
1. 输入暗号 → 对方不存在
2. 弹出选择："对方不在？你可以帮ta快速建一个临时画像，也能看到碰撞结果"
3. 用户回答 5 个关于朋友的简化问题：
   - "ta做决定的时候是更靠直觉还是更靠分析？"
   - "ta在群体中更倾向于表达还是观察？"
   - "ta处理冲突的方式是什么？"
   - "ta跟你在一起的时候，是ta主动联系你多还是你联系ta多？"
   - "用三个词描述ta的性格"
4. LLM 基于用户描述推测朋友的关键维度值（标记为"推测"，confidence 较低）
5. 生成碰撞结果（结果中标注"基于你对ta的描述，可能有偏差"）
6. 页面底部：生成一张"邀请卡"——"XXX，我觉得我们可能是【互补搭档型】，来做了你自己的画像看看准不准？"
```

关键设计：
- 推测结果**必须有偏差标注**——否则会让用户产生不切实际的期望
- 碰撞体验**是完整的**——用户能看到碰撞结果，不会因为朋友不在而卡住
- 邀请卡**是情感驱动的**——不是"来用这个产品"，而是"我想看看我说得对不对"

**模式 C：公开原型碰撞（单人 AI 对练）**

预置一系列"原型角色"，任何人都可以与之碰撞，不需要朋友。

```
原型角色库（12-15个）：
- "典型的回避型依恋者"：高回避、低焦虑、边界高壁垒、冲突冷处理
- "高开放性探索者"：直觉型思维、行动优先、自由倾向强、向往新身份
- "安全型连接者"：安全依恋、即时修复、质量时间型亲密语言、选择性社交能量
- "理想主义者"：抽象型思维、公平倾向强、未来导向、对话思考型
- ...（覆盖不同维度组合的代表性形象）
```

原型碰撞的价值：
- 单人用户有内容可消费
- 原型碰撞结果本身就是内容："你和你最应该小心的那种人是什么关系"
- 可以作为病毒传播素材："每个人都应该跟'回避型依恋者'碰撞一次"
- 驱动用户好奇自己的类型："我是哪种原型？"

#### 3.1.3 三种模式的关系

```
用户输入暗号
├─ 对方存在 → 模式A：双人完整碰撞
└─ 对方不存在
   ├─ 选择"我帮ta描述" → 模式B：单人推测碰撞 → 发邀请卡
   └─ 选择"看看你是哪种" → 引导对方来做自画像
                                ↓
                          或先玩模式C：和公开原型碰撞
```

### 3.2 关系分层：不同关系，不同分析

#### 3.2.1 为什么需要分层

同一对维度数据，在不同关系上下文里，关注点完全不同：

| 关系类型 | 用户想知道 | 分析侧重 |
|----------|-----------|----------|
| 恋人 | 我们为什么吵架？怎么更好地在一起？ | 依恋风格兼容性、冲突修复模式匹配、亲密语言是否对齐 |
| 密友 | 我们为什么这么合得来？ | 共鸣区、价值观光谱重合度、思维层面角色互补 |
| 家人 | 为什么总是卡在同一个地方？ | 成长层面对比（"你不想成为ta的样子"）、边界风格冲突、演化方向 |
| 同事 | 怎么合作更高效？ | 不确定性应对配合、信息处理模式互补、冲突处理风格 |
| 刚认识的人 | 这段关系有潜力吗？ | 碰撞点预警、关系潜力评估、摩擦预警 |

#### 3.2.2 方案：碰撞前的"关系定义"

在碰撞输入暗号后，增加一步：

```
┌──────────────────────────────────────┐
│  "TA是你的...？"                      │
│                                      │
│  ○ 恋人 / 伴侣                       │
│  ○ 密友                             │
│  ○ 家人（父母/兄弟姐妹）              │
│  ○ 同事 / 合作伙伴                   │
│  ○ 刚认识不久，想了解                 │
│  ○ 不确定，你帮我判断                 │
└──────────────────────────────────────┘
```

选择后，碰撞 prompt 注入对应的分析框架：

```typescript
const RELATIONSHIP_FRAMEWORKS = {

    outputEmphasis: '关注兼容性、潜在冲突模式和成长空间',

  close_friend: {

    focusDimensions: ['escape_direction', 'boundary_style', 'conflict_handling', 'attachment_style'],

    toneAdjustment: '更克制、更多"理解但不一定认同"的框架',
  },
  colleague: {
    focusDimensions: ['info_processing', 'uncertainty_response', 'conflict_handling', 'expression_thinking'],
    outputEmphasis: '关注工作方式配合、决策互补、沟通效率',
    toneAdjustment: '更专业、更多"配合"和"效率"语言',

  new_acquaintance: {
```

#### 3.2.3 结果展示的差异

不同关系类型的结果页，信息架构不同：

- **恋人**：依恋风格象限图 + 冲突修复模式对比 放在最上面。角色标签偏重情感和成长层。
- **密友**：共鸣区 放在最上面。角色标签偏重思维和关系动力学层。
- **同事**：碰撞点（工作方式差异）放在最上面。摩擦预警放在显眼位置。角色标签偏重行动和思维层。
- **家人**：边界风格对比 + 冲突模式 + "演化方向"对比放在最上面。语气更克制。

### 3.3 行动层：洞察之后的行为指引

#### 3.3.1 设计原则

行动指引不是建议，不是鸡汤，不是"多沟通"。它必须：
- **具体到场景**：不是"尝试更直接"，而是"下次对方说'我觉得不太好'时，问ta'具体是哪里不好'"
- **基于维度数据**：行动指引来自于两个人的具体维度差异
- **微小**：是一个动作，不是一个目标。用户不需要改变自己，只需要尝试一个新的方式
- **可验证**：用户试了之后可以回来反馈"有用/没用"

#### 3.3.2 碰撞结果中的行动层

每个碰撞结果新增一个区域：**"试试看"**，包含 2-3 个基于双方维度差异的具体行动建议。

行动建议的生成规则：

```typescript
// 规则引擎 + LLM 混合生成
function generateActionHints(myDims, theirDims, relationshipType, collisionResult) {
  const actions = []

  // 规则驱动的行动建议
  // 例1：如果你倾向"真实"，对方倾向"善意"
  if (myDims.values.truth_vs_kindness > 50 && theirDims.values.truth_vs_kindness < -20) {
    actions.push({
      scenario: "当你需要告诉对方一个不太好的消息时",
      action: "先说你观察到的具体事实，而不是直接给判断——给对方一个缓冲，ta需要这个空间",
      based_on: "你的真实倾向较高，对方更看重善意",
      expected_effect: "对方更可能接受你的反馈，而不是防御",
    })
  }

  // 例2：如果一方的冲突处理是"冷处理"，另一方是"即时修复"
  if (myDims.relationship_patterns.conflict_repair === '冷处理'
      && theirDims.relationship_patterns.conflict_repair === '即时修复') {
    actions.push({
      scenario: "当你们发生争执后",
      action: "告诉对方'我需要一点时间想想，大概明天早上我们聊'——给ta一个明确的时间，ta会安心很多",
      based_on: "你需要空间处理冲突，对方需要尽快修复",
      expected_effect: "对方不会在你沉默的时候焦虑，你也有了思考的时间",
    })
  }

  // LLM 补充生成 1-2 个规则覆盖不到的
  // ...
  return actions.slice(0, 4)
}
```

#### 3.3.3 自画像中的行动层

自画像结果页也需要行动层，但方向不同——碰撞的行动是"怎么和他相处"，自画像的行动是"怎么跟自己相处"。

在 `evolution_direction`（演化方向）的基础上，每个"未完成的自我"维度附带一个行动：

```
"被压抑的表达"：你有更直接、更锋利的一面，但还没稳定地放出来。
→ 试试看：下周有一次机会，你可以不说"都行"而是说"我想要X"。就一次。

"向往的身份"：你正在变成"温柔但更清楚"的人。
→ 试试看：下次你心里有判断但犹豫要不要说时，先问自己——"如果我不说，会后悔吗？"
```

#### 3.3.4 行动验证闭环

用户看到行动建议 → 尝试 → 回来反馈 → 反馈影响后续分析。

这不是一个"记录"功能，而是简单的反馈按钮：

```
每个行动建议下方：
  ┌──────────────────────────────┐
  │  你试过了吗？                  │
  │  ○ 试了，有用                │
  │  ○ 试了，不太适合我           │
  │  ○ 还没试                    │
  └──────────────────────────────┘
```

点击"试了，有用"→ 记录 + 增强该维度信号。
点击"试了，不太适合"→ 记录 + 下次生成行动时调整方向。
点击"还没试"→ 在 3 天后推送提醒："上次建议你试试的那个小改变，后来有尝试吗？"

### 3.4 留存：时间轴与变化记录

#### 3.4.1 留存逻辑

做完自画像 + 碰撞 3 个人之后，产品能持续提供的价值是什么？

答案：**变化本身就是价值。**

- 3 个月后重测，你的人格标签从"调和者"变成"边界建筑师"——这意味着什么？
- 你和同一个人 3 个月后的碰撞结果变了——你们的关系在变好还是在变差？
- 你连续做了 5 次碰撞，你可能开始看到自己在不同关系中的**重复模式**

#### 3.4.2 个人时间轴

```
┌────────────────────────────────────────────┐
│  你的演化时间线                              │
│                                            │
│  2026/02       2026/03       2026/05       │
│  「调和者」   → 「调和者」   → 「边界建筑师」 │
│  深度型         深度型         深度型        │
│  安全依恋       安全依恋       安全依恋      │
│                                            │
│  变化提示：                                 │
│  ⚡ 你的「冲突处理」从"调和"变成了"整合"——    │
│     你不再寻找折中，而是重构问题本身。        │
│  ⚡ 这和你在 3 月份换工作有关吗？             │
│                                            │
│  稳定性提示：                               │
│  🔒 你的「安全型依恋」一直很稳定——            │
│     这是你关系中最坚实的基底。               │
└────────────────────────────────────────────┘
```

**自动变化检测：**
- 当用户重测时，自动对比新旧结果
- 标注"显著变化的维度"（差异超过阈值）和"稳定的维度"
- 给出变化的可能解释——不一定是正确的，但引发用户思考
- 稳定的维度本身也是洞察——"有些东西是你不会变的底色"

#### 3.4.3 关系时间轴

```
和 [朋友名字] 的关系演化

2026/01          2026/04
【温和磨合型】 → 【互补搭档型】

变化：
→ 你们的碰撞点从 3 个减到 1 个
→ 共鸣区从 1 个增到 2 个
→ 角色从"刹车片"变成"副驾驶"

关系潜力：从"需要磨合"变成"可以一起做事"
```

这需要产品鼓励用户对同一个人做多次碰撞。触发方式：
- 碰撞记录页显示"上次碰撞于 1 月 15 日，看看有没有变化？"
- 如果某个维度的置信度有明显变化（重测后），提示"你的画像有一些变化，要和之前碰撞过的朋友重新看看吗？"

#### 3.4.4 跨关系模式发现

当用户有 3 次以上碰撞记录时，可以发现跨关系模式：

```
你在不同关系中的模式：

→ 你倾向于成为对方的「磨刀石」或「翻译器」——你在关系中常常是那个帮对方理清思路的人
→ 你的碰撞中最常出现的是"速度差"——你和多数人都在节奏上不同
→ 你最容易产生共鸣的维度是"深度vs广度"——你在寻找同样重视深度的人
```

这本身又是一个更高层次的"被理解感"——不只是看一段关系，而是看到自己在关系中的重复模式。

#### 3.4.5 反思触达（轻量留存手段）

不依赖用户主动回来，而是温和地触达：

- **每周**：一个反思问题。基于用户的`unfinished_self`或`evolution_direction`。"你上周有练习'更清楚地说出自己的想法'吗？这不需要一个巨大的场景——可能只是咖啡店点错单时说'我要的是拿铁不是美式'。"
- **每月**：一个模式检查。"上个月你说你正在学着更直接地表达。回头看看——有哪次你做到了吗？"
- **每季度**："要不要重新测一下？3 个月前的你是'调和者'，现在的你呢？"

这些触达**不是通知**（太烦人），而是在用户再次打开产品时展现。目的不是高频召回，而是让用户知道"这个产品没有忘了我"。

## 四、整体实施优先级

### 4.1 优先级原则

1. **先修体验断层，再加新功能**：当前的对话→分析→碰撞流程中有几处体验断裂（被理解感不足、过渡生硬），应该先修好再扩张
2. **先保证单人能跑通，再优化双人**：冷启动问题不解决，传播层没法验证
3. **先做表达层的升级，再做留存**：没有可传播的身份和关系，留存的"变化记录"就没有对比基线
4. **数据模型能承载未来**：每个阶段的实现都要考虑到后续阶段的扩展，避免返工

### 4.2 分阶段计划

---

**Phase 0：数据模型升级（所有阶段的前置，0.5-1 周）**

不涉及 UI 变更，只升级数据结构和 prompt：

1. `DimensionResult` 增加：
   - `identity_label`：身份标签（主标签 + 修饰标签）
   - `evidence_structured`：结构化证据（替代当前的一行 evidence）
2. `CollisionResult` 增加：
   - `relationship_type`：关系类型命名
   - `action_hints`：行动建议数组
   - `relationship_context`：关系类型（恋人/密友/家人等）
3. `UserRecord` 增加：
   - `analysis_versions`：分析版本历史数组
   - `collision_versions`：碰撞版本历史数组

---

**Phase 1：被理解感 + 证据层（2-3 周）**

这是用户旅程的最前端，也是信任的基础。先做。

**Phase 5：留存（时间轴 + 反思触达）（3-4 周）**

最后一个阶段。前面的功能让用户形成使用习惯和行为数据，时间轴才有内容可展示。

| 任务 | 涉及文件 | 工期 |
|------|----------|------|
| 分析版本管理：重测时保留历史版本，自动对比变化 | `analysis.ts` + `supabase.ts` | 2 天 |
| 碰撞版本管理：同一对关系的多次碰撞保留历史 | `collision.ts` + `supabase.ts` | 1 天 |
| 个人时间轴页面：展示身份标签演化 + 维度变化检测 | 新页面 | 4 天 |
| 关系时间轴：同一对关系的变化追踪 | `CollisionPage.tsx` 扩展 | 2 天 |
| 跨关系模式发现：3+ 碰撞后的模式总结 | 新模块 | 3 天 |
| 反思触达：每周反思问题 + 行动反馈追踪 | 新模块 + prompt | 2 天 |

**Phase 5 的验收标准：**
- 用户重测后能看到与历史版本的对比
- 同一对关系多次碰撞能看到变化趋势
- 用户有 3+ 次碰撞记录后能看到跨关系模式总结

---

### 4.3 总结：为什么这个顺序

```
Phase 0 (数据模型)
   ↓
Phase 1 (被理解感 + 证据层)
   ↓  先修体验断层，让用户愿意走完流程
Phase 2 (冷启动 + 单人体验)
   ↓  保证新用户在任何情况下都能跑通核心闭环
Phase 3 (碰撞表达升级)
   ↓  让碰撞结果有传播力
Phase 4 (行动层)
   ↓  建立产品差异化
Phase 5 (留存)
   ↓  在以上基础上建立长期价值
```

每个阶段都建立在前一个阶段的基础上，且每个阶段结束时产品都是**完整可用的**——不是一个"做了 20% 的社交功能"，而是一个"没做社交但体验已经很好的产品"。

**MVP 分界线建议：** Phase 1 + Phase 2 完成时即可对外发布。Phase 3-5 可以基于真实用户数据验证假设后逐步迭代。
