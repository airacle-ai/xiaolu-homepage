# Oprah 产品 0→1 分析

## 当前状态诊断

**已有的东西：**
- 一套能跑通的对话→分析→碰撞的技术骨架
- 19 维度的人格模型（思维方式 / 价值观 / 关系模式 / 未完成的自我）
- 4 位暗号作为"可分享的身份标识"
- 28 个角色标签的关系碰撞系统（5 层面 × 非对称双方向）
- 置信度提升机制（refinement loop）

**缺的东西（按优先级）：**

| 缺口 | 影响 |
|------|------|
| 结果缺乏"可传播的身份符号" | 用户做完不知道怎么跟朋友说"我是XX型" |
| 证据层不可见 | 用户只看到结论，看不到"凭什么这么说"，信任感弱 |
| 问题库耦合在 prompt 里 | 无法做版本管理、A/B 测试、多语言 |
| 碰撞结果缺少"一句话总结" | 分享出去只有暗号，没有钩子 |
| 没有"关系类型命名" | 碰撞分析很细但缺一个顶层标签（如"镜像成长型"） |
| 简化/详细两版尚未实现 | 不同场景的用户用同一套流程 |

---

## 一、产品定位与北极星指标

**一句话定位：** AI 对话式自画像 + 关系碰撞工具。用户通过和 AI 聊天获得一个"人格身份"，然后用这个身份和朋友的身份碰撞，看到彼此之间独特的化学反应。

**核心价值主张：**
- 自我认知：比 MBTI 更立体，比心理咨询更轻量
- 关系理解：不是"你们合不合"，而是"你们之间有什么"

**北极星指标：** 周活跃碰撞对数（一个人发起碰撞 + 对方接受/回应 = 1 对）

**为什么是碰撞而不是自画像？** 因为自画像是一次性消费，碰撞是持续消费 + 社交传播的引擎。一个人做完自画像后，如果没有碰撞场景，7 天内必然流失。但每次碰撞都是一次"被看见"的体验，会驱动双方都回来。

## 二、用户生命周期与关键路径

### 2.1 获客：用户怎么来

```
朋友分享「关系名片」→ 接收方看到"你和XX的关系是【镜像成长型】"
→ 点击进入 → 需要先完成自己的自画像 → 自然转化为新用户
```

**关键设计：** 碰撞结果必须是一张可分享的卡（不是一段文字），卡上包含：
- 双方的身份标签（不是暗号，是可读的标签）
- 关系类型命名
- 一句最精彩的洞察
- 扫码/链接入口

### 2.2 激活：第一次"aha moment"

当前 demo 的 aha moment 太晚——用户要聊完 15 轮才能看到结果。需要前置两个轻量 aha：

1. **对话中即时洞察**（已有雏形）：Oprah 说出"你刚才说你很重视自由，但在前面那个场景里你选择了留在群体里"——这类矛盾洞察需要在第 3-4 轮就出现。
2. **即时维度解锁动画**：每覆盖一个新维度，星星点亮 + 维度名称浮现，让用户感觉"我在被拼出来"。

### 2.3 留存：为什么用户要回来

- 碰撞通知："XXX 输入了你的暗号，想看看你们的关系"
- 新维度解锁：如果第一次只覆盖了 12/19 维，邀请"继续探索剩下的自己"
- 关系变化：同一个朋友，3 个月后再碰撞，结果可能不同

### 2.4 传播：用户分享什么

**当前问题：** 4 位暗号（如 "KXRP"）本身没有信息量，无法激发好奇。

**需要变成：** 一个**身份标签**（如"调和者·深度型·安全依恋"），搭配关系碰撞的**关系命名**（如"镜像成长型"），组成一张可传播的关系名片。

## 三、产品架构：从单次对话到身份系统

### 3.1 三层架构

```
┌──────────────────────────────────────────┐
│  第三层：关系碰撞（传播层）                │
│  - 关系类型命名（如"镜像成长型"）          │
│  - 非对称角色卡（你之于对方 / 对方之于你） │
│  - 一张可分享的关系名片                    │
└──────────────┬───────────────────────────┘
               │ 输入：两个人的身份标签 + 维度数据
┌──────────────┴───────────────────────────┐
│  第二层：身份标签（表达层）                │
│  - 顶层身份命名（如"调和者·深度型"）       │
│  - 一句话画像（可复制传播）                │
│  - 证据层（为什么是这个标签）              │
└──────────────┬───────────────────────────┘
               │ 输入：19 维结构化数据
┌──────────────┴───────────────────────────┐
│  第一层：AI 对话（采集层）                 │
│  - 简洁版：6-8 轮，覆盖 8-10 个核心维度    │
│  - 详细版：15-20 轮，覆盖 19 个全维度      │
│  - 置信度提升：定向补问                    │
└──────────────────────────────────────────┘
```

### 3.2 简洁版 vs 详细版的设计差异

| 维度 | 简洁版 | 详细版 |
|------|--------|--------|
| 轮数 | 6-8 轮 | 15-20 轮 |
| 覆盖维度 | 8-10 个核心维 | 全部 19 维 |
| 情境题 | 4 个高信号场景 | 8-10 个全覆盖场景 |
| 结果 | 身份标签 + 一句话画像 | 完整 19 维地图 + 标签 |
| 可碰撞 | 可以（用简化维） | 可以（全维） |
| 适用场景 | 首次体验 / 朋友邀请后快速生成身份 | 深度自我探索 |
| 入口文案 | "快速了解自己，3 分钟" | "深入探索，像照镜子一样看自己" |

**关键设计决策：** 简洁版产生的结果格式和详细版是**兼容**的。简洁版未覆盖的维度标记为"未探索"，碰撞时只用共同覆盖的维度计算。用户从简洁版升级到详细版时，保留已有维度的结果，只补充未覆盖的。

### 3.3 从简洁版→详细版的升级路径

```
用户完成简洁版 → 看到身份标签和碰撞 → 朋友碰撞结果里出现一个ta没覆盖的维度
→ "对方在这个维度上和你很不同，但你还未探索这个维度" → 点击进入详细版补完
```

这不是"升级付费"逻辑，而是**社交驱动的内容解锁**——最好的转化钩子不是产品本身的提示，而是朋友碰撞结果里的空白。

## 四、问题库结构化（为版本化做准备）

### 4.1 当前问题

问题库硬编码在 `COACH_SYSTEM_PROMPT` 里（约 3000 字），LLM 自行选择出题。这导致：
- 无法做 A/B 测试（两个版本的问题库无法并行对比）
- 无法根据用户画像动态调整题目难度/方向
- 无法做多语言（prompt 是中文耦合的）
- 修改一个问题需要重新部署整个 prompt

### 4.2 目标结构

```typescript
// src/lib/questionBank.ts

interface Question {
  id: string                    // "Q001"
  version: number               // 方便迭代
  category: 'core' | 'extended' // core = 简洁版必选, extended = 详细版扩展
  text: string                  // 情境题文本
  dimensions: DimensionKey[]    // 主要覆盖的维度
  followUpDirections: {         // 追问方向（按用户回答类型分支）
    trigger: string             // 用户回答类型："avoidant" | "engaged" | "contradictory" | "surface"
    questions: string[]         // 追问问题列表
  }[]
  conditions?: {                // 出题前置条件（可选）
    requireDimensions?: DimensionKey[]  // 需要这些维度已覆盖才出
    excludeDimensions?: DimensionKey[] // 已有强信号则跳过
    priority: number            // 优先级 1-5
  }
}
```

### 4.3 问题选择策略

不依赖 LLM 自行选择，改为**前端 planner** 选题：

```
1. 追踪每个维度的信号强度（由 LLM 每轮标记 DIMS）
2. 从 questionBank 中筛选"覆盖弱信号维度最多"的题目
3. 如果有多个候选，按 priority 排序
4. 将选中的题目以 system message 形式注入对话
```

这样做的好处：
- 题目选择逻辑可测试、可调优
- 同一用户可以重测（换一批题）
- 支持"跳过某题"（用户不想回答某个情境）
- A/B 测试只需换 questionBank 引用

## 五、身份标签系统（解决"表达符号不够强"）

### 5.1 当前问题

当前结果是一张"自我地图"——19 个维度的罗列展示，用户得不到一个**能一句话说出去的标签**。对比 MBTI 的成功：4 个字母 + 一个名字（如"提倡者 INFJ"），用户可以轻松地说"我是 INFJ"。

Oprah 的 19 维模型比 MBTI 更丰富，但当前缺乏一个**顶层压缩**机制。

### 5.2 设计：三层身份标签

```
第一层：核心身份（1 个词或短语，用于社交传播）
  例："调和者"、"边界建筑师"、"温柔的反叛者"
  生成方式：由 LLM 根据 19 维结果，从标签库中选择或原创

第二层：修饰标签（2-3 个词，对核心身份的展开）
  例："深度型 + 安全依恋"
  来自 values 和 relationship_patterns 的关键维度

第三层：一句话自述（用于名片、分享卡）
  例："一个认真处理关系温度的人，正在学着更清楚地表达自己。"
  来自 overall_portrait 的精简
```

完整身份表达：**「调和者 · 深度型 · 安全依恋」**

### 5.3 身份标签的生成逻辑

不靠 LLM 自由发挥，而是规则 + LLM 结合：

```typescript
// 规则层：从维度的确定值映射到标签候选
const CORE_IDENTITY_RULES = {
  // conflict_handling + boundary_style 的组合
  '调和_选择性开放': '边界建筑师',
  '整合_高壁垒': '温柔的反叛者',
  '回避_渐进开放': '沉静的观察者',
  // ...
}

// LLM 层：规则无匹配时，或用户选择"自定义模式"时，由 LLM 生成
```

每个身份标签附带：
- **为什么是这个标签**（证据摘要，从 evidence 字段提取）
- **和你相似的人**（模糊匹配，展示"有 342 人也拥有'调和者'标签"）

## 六、证据层设计（"为什么"）

### 6.1 当前问题

每个维度有 `evidence` 字段（LLM 生成的一句话证据），但：
- 它藏在 MapPage 的卡片里，需要用户点击展开才能看到
- 它是后台字段，不是前台叙事
- 它和 insight 混在一起，用户不知道哪个是"证据"哪个是"解读"

### 6.2 证据层的三个展示层级

```
┌─────────────────────────────────────────────┐
│  层级 1：一句话证据（地图卡片内）             │
│  "你面对深夜求助时先评估关系，再决定付出——     │
│   这个模式在 3 个不同场景中一致出现。"         │
│  展示方式：每个维度卡片默认显示，不折叠        │
└─────────────────────────────────────────────┘
                    ↓ 点击展开
┌─────────────────────────────────────────────┐
│  层级 2：具体引用（弹窗/展开区）              │
│  - 情境 1：「深夜的求助」中你选择先确认再回应  │
│  - 情境 2：「朋友的危险决定」中你先分析风险    │
│  - 情境 3：「选择恐惧」中你列出选项再比较      │
│  每个引用带原始对话上下文（你的原话 + Oprah 追问│
└─────────────────────────────────────────────┘
                    ↓ 点击"质疑"
┌─────────────────────────────────────────────┐
│  层级 3：反例检查（置信度提升入口）            │
│  "但你真的在所有关系里都这样吗？              │
│   有没有人对你来说是例外？                     │
│   有没有一个情境你会选择直接行动？"            │
│  用户可以进一步探索，修正判断                  │
└─────────────────────────────────────────────┘
```

### 6.3 evidence 的数据结构升级

当前 evidence 是 LLM 生成的一句话字符串，需要升级为结构化引用：

```typescript
interface Evidence {
  summary: string                    // 一句话总结
  quotes: {                          // 具体引用
    turnIndex: number                // 对话轮次
    userSaid: string                 // 用户原话摘要
    situation: string                // 当前情境名
    signal: string                   // 这段话暴露了什么信号
  }[]
  consistency: 'cross_situational' | 'single_situation' | 'inferred'
  // 跨情境一致 / 单情境 / 间接推断
}
```

这需要在 `analyzeDimensions` 的 prompt 中要求 LLM 输出结构化 evidence，而不仅是一句话。同时 MapPage 需要新增引用展示组件。

## 七、关系碰撞的表达升级

### 7.1 当前问题

碰撞结果很丰富（5 角色 × 双方向 + 碰撞点 + 共鸣区 + 摩擦预警），但：
- 缺少一个顶层命名——用户无法说"我们的关系是XX型"
- 缺少一张可截图分享的卡
- 角色标签（如"磨刀石"）很好但需要解释才能理解

### 7.2 关系类型命名

在 `collision_prompt` 中增加输出：`relationship_type`，是一个简短的关系命名，用于社交传播。

类型命名候选库：

| 类型 | 触发条件 | 含义 |
|------|----------|------|
| 镜像成长型 | 相似度高 + 共鸣区 ≥ 2 | 你们太像了，互相映照 |
| 互补搭档型 | 碰撞点 ≥ 3 + 角色互补 | 你缺的正好是ta有的 |
| 温和磨合型 | 摩擦预警中等 + 共鸣区 ≥ 1 | 有差异但互相愿意理解 |
| 单向支撑型 | 两个方向的角色强度差 ≥ 2 级 | 一方给的多，一方接收的多 |
| 启发者型 | 思维层角色强 + 成长层角色强 | 你们在互相推动对方变好 |
| 安全基地型 | 情感层角色强 + 依恋风格兼容 | 在一起就是充电 |
| 挑战者型 | 碰撞点 ≥ 3 + 摩擦预警高 | 你们会争执但也会成长 |

### 7.3 可分享的关系名片

碰撞结果页增加一个"生成名片"按钮，产出一张包含以下信息的卡片图：

```
┌──────────────────────────────────┐
│         Oprah 关系名片            │
│                                  │
│   [你的身份标签]  ×  [朋友身份标签] │
│                                  │
│      ✨ 【镜像成长型】 ✨          │
│                                  │
│  "一段适合一起把想法打磨成行动     │
│   的关系。"                      │
│                                  │
│  对方是你的：开窗人 · 充电宝       │
│  你是对方的：磨刀石 · 安全着陆点   │
│                                  │
│  [二维码/链接 进入 Oprah]         │
└──────────────────────────────────┘
```

## 八、实施路线图（分阶段）

### Phase 1：身份标签 + 证据层（2-3 周）

**目标：** 让做完自画像的用户有一个"可传播的身份"，且能看懂为什么。

1. 升级 `analyzeDimensions` prompt，增加身份标签生成（在 `DimensionResult` 中新增 `identity_label` 字段）
2. 升级 `schemas.ts` 的 `analysisToolSchema`，增加 `identity_label` 和结构化 `evidence`
3. 改造 `MapPage.tsx` 顶部：身份标签作为 Hero 区域，替换当前的暗号区域
4. 新增 `EvidencePanel` 组件：展示证据引用
5. 保留暗号（用于碰撞输入），但不再作为主视觉元素

### Phase 2：简洁版 + 问题库结构化（3-4 周）

**目标：** 降低新用户门槛，为后续迭代打基础。

1. 创建 `src/lib/questionBank.ts`，将 15 个情境题迁移为结构化数据
2. 创建 `src/lib/planner.ts`，实现维度追踪 + 题目选择逻辑
3. 在登录后增加**模式选择**：简洁版（3 分钟）vs 详细版（10 分钟）
4. 修改 `COACH_SYSTEM_PROMPT`：不再包含题目文本，改为接收 planner 注入的题目
5. 确保简洁版和详细版的结果格式兼容

### Phase 3：碰撞表达升级 + 分享传播（3-4 周）

**目标：** 让碰撞结果可传播，驱动自然增长。

1. 升级 `collision_prompt`，增加 `relationship_type` 命名
2. 设计并实现"关系名片"生成（Canvas 渲染 → 保存图片）
3. 实现分享流程：名片图片 + 邀请链接 + 暗号
4. 碰撞历史页面增加筛选和搜索

### Phase 4：社交功能 + 留存（4-6 周）

**目标：** 让用户持续回来。

1. 碰撞通知："有人想看看和你的关系"
2. 维度探索进度追踪："你还有 7 个维度未解锁"
3. 关系变化追踪：同一对关系 3 个月后的变化
4. 身份标签社区：同标签用户的聚合统计（匿名化）

## 九、关键风险与假设验证

### 9.1 核心假设

| 假设 | 验证方式 | 失败信号 |
|------|----------|----------|
| 用户愿意聊 6-15 轮来完成测试 | 简洁版完成率 > 60% | 完成率 < 30%，需要缩短到 4 轮 |
| 4 位暗号能驱动碰撞行为 | 暗号被分享的比例 > 20% | 暗号生成后 7 天内无碰撞记录 |
| 碰撞结果有传播力 | 名片保存/分享率 > 15% | 无人截图分享 |
| 身份标签能成为社交货币 | 用户在社交媒体提及标签 | 标签只在产品内出现 |

### 9.2 技术风险

- LLM 输出稳定性：结构化 evidence 和 identity_label 需要 JSON 模式严格约束，否则容易出现格式漂移
- 成本：详细版 15-20 轮对话 + 分析 + 碰撞，每次用户会话的 token 消耗较大。需要在 prompt 长度和模型选择上做平衡
- 编码问题：文档提到源码中文是乱码（mojibake），需要确保编辑时使用正确的 UTF-8 编码

### 9.3 体验风险

- **过度分析感：** 如果 Oprah 每轮都给洞察，用户会觉得被审视。洞察频率需要控制。
- **标签不准的尴尬：** 如果用户觉得身份标签"不对"，会产生负面体验。需要有"我不认同，重新分析"的出口。
- **碰撞隐私：** 两个人的维度数据互相暴露可能造成社交尴尬。需要控制碰撞结果暴露的粒度——只暴露和对方相关的维度对比，不暴露完整画像。

## 十、总结：一句话说清产品

**Oprah 是一个 AI 对话式身份系统。你和 AI 聊天获得一个"身份标签"，然后输入朋友的标签，看到你们之间独特的化学反应——不是测性格，是理解关系。**

三个关键词：
1. **身份标签**（而不是暗号）——解决表达和传播
2. **证据可见**（而不只是结论）——解决信任
3. **关系命名**（而不只是碰撞数据）——解决"这有什么用"
